# speckit.plan

Generate or update `specs/[FEATURE_DIR]/plan.md` — the structural plan for the entire node graph.

## User Input

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty).

Optional flags:
- `--update` � revise existing flowmap without overwriting approved sections
- `--act [N]` � regenerate a specific act only

## Pre-Execution Checks

**Check for extension hooks (before planning)**:
- Check if `.speckit/extensions.yml` exists in the project root.
- If it exists, read it and look for entries under the `hooks.before_plan` key.
- Process as standard hook block (Optional/Mandatory). Skip silently if absent.

**Document checks**:
1. Confirm `specs/[FEATURE_DIR]/spec.md` and `.specify/memory/constitution.md` exist.
2. Check `.specify/memory/constitution.md` has no unresolved `[NEEDS CLARIFICATION]` markers � warn for any found.
3. If `specs/[FEATURE_DIR]/plan.md` exists and `--update` not set: ask user to confirm overwrite.

## Outline

> **CONTENT PRE-FILL MANDATE � READ BEFORE EXECUTING ANY STEP**
> Every file generated by this command (plan.md, variables.md, mechanics.md, character profiles, endings.md, glossary.md, world-building.md) MUST be written with actual game-specific content drawn from `spec.md` and `constitution.md`. Bare template copies with unfilled `[placeholder]` markers are a **command failure**. The test: every file must be immediately useful to the designer — no blank scaffolds, no unresolved boilerplate. Only write `[NEEDS CLARIFICATION: reason]` when a field genuinely cannot be inferred from available documents.

1. **Setup**:
   - Ensure the directory `specs/[FEATURE_DIR]/` exists.
   - Locate `plan-template.md` using the preset template resolution order
   - Copy it to `specs/[FEATURE_DIR]/plan.md`
   - This becomes the working file � all subsequent steps fill it with actual content

2. **Load context**: Read `specs/[FEATURE_DIR]/spec.md` and `.specify/memory/constitution.md`.

3. **Fill Narrative Parameters** in `specs/[FEATURE_DIR]/plan.md`:
   - Extract: genre, engine target, player perspective, estimated play time, target audience, series position, tone, game bible version from the spec files
   - Mark unknowns as `[NEEDS CLARIFICATION]`

4. **Game Bible Check**:
   - Evaluate all gates in the Game Bible Check table in `specs/[FEATURE_DIR]/plan.md`
   - Mark each ? PASS, ?? NEEDS WORK, or ? VIOLATION
   - A ? VIOLATION blocks Phase 0 unless the user provides explicit justification � document in the Justified Violations table

**Phase 0 � Supporting Documents**:

> **Pre-fill rule**: Every supporting document MUST be written with actual game-specific content inferred from `spec.md` and `constitution.md`. Replace every `[placeholder]` marker with real content. Only write `[NEEDS CLARIFICATION: reason]` when a field genuinely cannot be inferred.
> **Language Rule**: All supporting documents (plan.md, variables.md, mechanics.md, character profiles, endings.md, glossary.md, world-building.md) MUST be generated in English (`en`) by default, regardless of any `[LANGUAGE]` setting in the constitution.

- Generate `specs/[FEATURE_DIR]/variables.md`: register every state variable identified across the narrative design doc and game bible � variable name, type, default value, description, which nodes read/write it, which endings gate on it. Group by category: trust, flags, inventory, attributes, counters.

- Generate `specs/[FEATURE_DIR]/mechanics.md`: one schema entry per mechanic type active in the game bible. For each:
  - Mechanic name, tier (1/2), hook syntax (engine-specific)
  - Trigger condition, effect, failure state
  - Which nodes use it (populated from narrative design doc Key Nodes list)
  - Cross-reference to `specs/[FEATURE_DIR]/variables.md` entry

- Generate character profiles: one file per key NPC at `specs/[FEATURE_DIR]/characters/[npc-name].md` using `characters-template.md` (use the character's name in lowercase-kebab-case, e.g., `specs/[FEATURE_DIR]/characters/mira-voss.md`). For each character, populate all sections:
  - **I. Core Identity**: dominant trait, strength, flaw, drive, fear, background foundation, internal arc (wound / want / need / transformation ? range of ending states)
  - **II. Speech Patterns & Voice**: vocabulary register, recurring word clusters, intensifiers, contractions, example vocabulary pool (including words they avoid), dialogue style (directness, power dynamic, deflection strategy, under-pressure behavior, subtext pattern), 3 sample dialogue exchanges with tone + subtext across trust levels
  - **III. Internal Monologue**: cognitive mode, blind spot, self-deception pattern, obsessive loop, 3 voice examples (discovery / conflict / decision)
  - **IV. Physical Expression**: micro-obsession (must recur across nodes), stress tell, comfort behavior, authority posture, signature appearance detail
  - **V. Emotional Landscape**: 4�6 primary emotions with triggers, visible behaviors, and what each masks; default channel; overflow and recovery patterns; 2+ moments of real doubt
  - **VI. Relationship to Player & Other Characters**: player dynamic (communication pattern, subtext, the repeating loop, trust variable), plus one entry per significant NPC relationship
  - **VII. Trust State Thresholds**: score ranges, state labels, dialogue register, choices unlocked per level; starting trust value
  - **VIII. Dialogue Register by Trust State**: how Section II voice sounds at each trust level (hostile / cautious / neutral / friendly / ally), with a sample line per level
  - **IX. NPC State Machine**: all valid states, Ink integer mapping, triggers per state
  - **X. Bark Lines**: greeting, reaction, and ambient barks consistent with Section II voice
  - **XI. Branch Behavior Map**: choices unlocked/removed, content replaced, information revealed per condition
  - **XII. Knowledge State**: what this character knows, when they knew it, what condition gates the player learning it
  - **XIII. Scene Guidance**: nodes this character drives, sensory signature, emotional arc progression table per act
  - **XIV. Verbal Tics**: repeated phrases with signaled states, speech pattern under stress, the sentence they never say aloud
  - **XV. Summary for Writers**: core challenge, default tone, variance points, Voice Homogeneity Test sentence, success indicator
  - **XVI. Series Arc State**: (non-standalone only) closing trust range, physical state, knowledge state, arc position, emotional state per game entry
  - Mark any field that cannot be completed without user input as `[NEEDS CLARIFICATION]`; do not invent details that contradict the narrative design doc

- Generate `specs/[FEATURE_DIR]/characters.md` as an index: register every character file, populate the voice homogeneity note, relationship map, and trust variable registry.

- Generate `specs/[FEATURE_DIR]/endings.md`: one entry per planned ending. For each ending:
  - Ending ID (END-A, END-B, etc.), type (`good` / `bad` / `neutral` / `secret` / `true`)
  - Gate conditions (variable checks, flag states)
  - Thematic statement this ending makes
  - Final node ID that leads to it
  - Emotional register of the final scene

- Generate `specs/[FEATURE_DIR]/world-building.md`: world rules, internal logic constraints, location entries (one block per named location with atmosphere, sensory anchors, nodes where it appears), and any faction or social system rules.

- Generate `specs/[FEATURE_DIR]/glossary.md` from `glossary-template.md`: seed with all invented terms, proper nouns, and variable names already present in `spec.md` and `constitution.md`. One entry per term; mark fields requiring more context as `[NEEDS CLARIFICATION]`.

- Generate `specs/[FEATURE_DIR]/themes.md` from `themes-template.md`: populate the Thematic Core from the `spec.md` dramatic question and theme statements. Seed the Motif Registry with any recurring objects, images, or phrases already present in the spec.md Key Items or world-building. Seed the Ending Thematic Divergence table from `specs/[FEATURE_DIR]/endings.md` thematic statements. Mark all underpopulated fields as `[NEEDS CLARIFICATION]`.

- Generate `specs/[FEATURE_DIR]/relationships.md` from `relationships-template.md` **only if** two or more key NPCs have intersecting trust variables or shared nodes in the spec: one REL-NNN block per significant NPC?NPC dynamic. Populate Relationship Brief from character profiles and spec.md. Mark all five key beats as `[NEEDS NODE]` if node IDs are not yet assigned. Skip silently if only one NPC is present.

- **Timeline** (conditional): if `spec.md` Temporal Parameters indicate non-linear discovery, a historical setting, or timer mechanics are declared in `constitution.md`: generate `specs/[FEATURE_DIR]/timeline.md` from `timeline-template.md`. Populate backstory events from the spec. Mark game-period events as `[NEEDS NODE]`. If none of these conditions apply, skip silently.

- **Series bible** (non-standalone games only): read Series Position from `spec.md`.
  - If anything other than `standalone`:
    - Check whether `specs/[FEATURE_DIR]/series-bible.md` already exists.
    - If it **does not exist**: generate a skeleton `specs/[FEATURE_DIR]/series-bible.md` from `series-bible-template.md`. Populate what can be inferred (series title, world rules, named entities, the current game's entry in the Games in Series table, carry-over variable registry, dramatic question if stated). Leave all fields requiring future games as `[TBD]`. Emit: `?? specs/[FEATURE_DIR]/series-bible.md created � review [TBD] entries before planning the next entry.`
    - If it **already exists**: load it and verify the current game has an entry in the Games in Series table. If missing, add the row. Check the Known Contradictions table; if any `OPEN` rows reference this game, emit a warning listing each `SX-NNN` code.
  - If `standalone`: skip.

**Phase 1 � Act Structure & Node Graph**:

5. **Fill Act Structure**:
   - Extract act count, branch model, approximate node count from narrative design doc
   - For each act: node count, key convergence node ID, dramatic purpose
   - Map all key scenes/nodes from the narrative design doc to act slots
   - Identify gaps (acts with no mapped nodes) and add `[NEEDS NODE]` notes
   - Fill the Open Narrative Threads table from the spec.md Key Items in `specs/[FEATURE_DIR]/plan.md`

6. **Generate Node Graph** (text representation):
   - One entry per planned node, grouped by act
   - **Node-type flag**: Mark each node as `(dialogue-centric)`, `(action)`, or `(mixed)` based on its primary interaction mode
     - `(dialogue-centric)` = multi-party conversation, player dialogue choices, NPC exchanges
     - `(action)` = player performs an action, investigates, etc.; NPCs react but no dialogue branching
     - `(mixed)` = both dialogue options and action consequences in the same node
   - Format: `NODE-ID: [name] (dialogue-centric|action|mixed) • [choice text] NODE-ID | [choice text] NODE-ID`
   - Show variable-gated branches: `? [choice � requires $var >= N] NODE-ID`
   - Identify all convergence points (merge nodes)

7. **Fill Information Asymmetry Map**: at each act boundary, what the player knows vs. what the narrative knows. Flag any branch that assumes knowledge the player cannot have reached.

8. **Branch Health Check**: identify dead ends, unreachable nodes, variables read before set, endings with no reachable path, and non-terminal nodes with only one choice.

9. **Pacing Overview**: estimate decision density, average tension score, and mechanic trigger count per act.

**Phase 1 � Re-evaluate Game Bible Check** after Phase 1 is complete.

**Phase 2 � Node Outline**:

10. **Generate the full node outline** in the `## Node Outline` section of `specs/[FEATURE_DIR]/plan.md`.

   **Node ID format**: `NODE-{act}{sequential_3digit}_{ShortName}` � e.g., `NODE-1001_Awakening`, `NODE-2003_BetrayalRevealed`. Endings use `END-{letter}_{ShortName}` � e.g., `END-A_TrueReconciliation`. Insertion nodes use letter suffix: `NODE-2003b` between `NODE-2003` and `NODE-2004`.

   For each node:
   - Assign a Node ID and short name (2�4 words, PascalCase)
   - Fill all fields: estimated word count, act, branch type (choice / converge / gate / terminal), player perspective (if switching), location, variables read, variables written
   - Write: opening hook (sensory, not summary), key beats (3�5 micro-events in causal order), choice architecture (each choice: text, condition, target node), NPC presence and emotional state, thematic work (how theme is carried without being stated), closing beat (what has changed, what the player now knows)
   - Flag narrative obligations from Key Scenes: mark nodes that fulfill a required beat with `[REQUIRED: fulfills KEY-SCENE-N]`

11. **Stop and report**: Report: `specs/[FEATURE_DIR]/plan.md` path, total nodes generated, total endings mapped, branch health status, any remaining `[NEEDS CLARIFICATION]` markers, open threads count.

12. **Build search index** (large projects):
    - Check whether `scripts/python/index.py` exists.
    - If it exists, run: `python scripts/python/index.py build` from the project root.
    - This indexes all supporting documents generated in Phase 0 for semantic search during drafting.
    - If the script is absent or fails, skip silently and note: `?? Search index not built � run python scripts/python/index.py build manually when ready.`

13. **Check for extension hooks** (after planning): check `hooks.after_plan` and process as standard hook block.
