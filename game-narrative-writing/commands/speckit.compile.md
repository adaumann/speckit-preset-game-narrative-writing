# speckit.compile

Compile node files to a playable output file in `specs/<specname>/output/<ENGINE>/`.

**Source priority**: prefers `specs/<specname>/export/<ENGINE>/` (if `speckit.export` has been run), falls back to `specs/<specname>/draft/<ENGINE>/`.

**Pipeline**:
1. Combine all source files and convert `[MECHANIC:...]` tokens to engine-native syntax → `compile.twee`
2. Run postprocessing scripts on `compile.twee` (inject headers, animations, theme overrides)
3. Compile with native compiler (tweego for SugarCube, inklecate for Ink)
4. Run postprocessing scripts on the compiled output

Uses native compilers (tweego for SugarCube, inklecate for Ink) with automatic retry logic for compilation errors.

**Output formats**:
- **Twine/SugarCube** (twee) → HTML file with StoryData JSON header (via tweego.exe)
- **Ink** (.ink) → Compiled JSON + HTML wrapper (via inklecate.exe)
- **Renpy** (.rpy) → Python script for Ren'py engine
- **Generic** (markdown) → Linked HTML pages

**Generic draft support**: `.md` files with `[MECHANIC:...]` tokens are automatically detected, converted to engine-native syntax, and wrapped in proper Twee passage headers — no separate `speckit.export` step required.

## User Input

```text
$ARGUMENTS
```

Accepted arguments:
- `[ENGINE]` - compile for a specific engine (e.g., `sugarcube`, `ink`, `renpy`, `generic`)
- `--all-engines` - compile all engines in export_engines configuration simultaneously
- `--force-rebuild` - force full rebuild even if output exists
- `--dry-run` - validate without writing output files
- `--walkthrough` - run the full-story walkthrough (BFS traversal of all branches) after compilation
- `--no-walkthrough` - skip runtime testing entirely (implies `--no-test`)
- *(no argument)* - compile to the engine specified in `constitution.md`

## Pre-Execution Checks

**Verify source files exist** (checks in order):
- Scan `specs/<specname>/export/<ENGINE>/` for all engine files (boilerplate + nodes from `speckit.export`)
- If no export dir, scan `specs/<specname>/draft/<ENGINE>/` for `story.twee` or `NODE-*.twee` files
- If no files found - halt: "No source files found for engine `$ENGINE`. Run `speckit.implement` (or `speckit.export`) first."

**Verify engine configuration**:
- Check `constitution.md` for `export_engines`
- Check `variables.md` for export names for the target engine
- Check `mechanics.md` for Tier 1 hook compatibility with target engine

## Execution Steps

### 1. Structural Validation (automatic)
- Parse all node YAML headers
- Validate required fields (node_id, title, status)
- Check mechanic hook syntax ([MECHANIC:...] blocks)
- Check choice block syntax ([CHOICE:...] blocks)
- Report all validation errors with line context

### 2. Load Node Files
- Read validated node files into memory
- Parse YAML headers and body content
- Handle both individual NODE files and combined story files

### 2b. Embed Illustration Images

**Illustration image setup (manual / external process)**:
- Illustration PNGs are **not generated by this tool**. You must create them externally (Midjourney, DALL-E, Stable Diffusion, commissioned artist, etc.).
- **Directory**: Place PNG files in `specs/[FEATURE_DIR]/assets/illustrations/`. Create the directory if it does not exist.
- **Naming convention**: Each PNG must be named exactly `<NODE_ID>.png` (e.g., `NODE-1001.png`, `END-A_Blessed.png`, `LOC-001.png`). The filename (without extension) must match the node_id exactly — case-sensitive.
- **Only PNG format is supported**. Other formats (jpg, webp, gif) are ignored.
- Start with key scenes (act openings, endings, major locations) rather than every node.

After loading node files, if `illustrations_enabled: yes` in constitution.md:
- Check `specs/[FEATURE_DIR]/assets/illustrations/` for PNG files matching node IDs (`<NODE_ID>.png`)
- For each node with a matching image file, embed it at the beginning of the node body content (after the YAML header):

  **SugarCube**: `[img[specs/[FEATURE_DIR]/assets/illustrations/<NODE_ID>.png]]`
  ```twee
  :: NODE_001
  [img[specs/[FEATURE_DIR]/assets/illustrations/NODE_001.png]]
  
  [node prose body...]
  ```

  **Ink**: `<img>` tag in the HTML wrapper or inline HTML in Ink:
  ```ink
  <img src="specs/[FEATURE_DIR]/assets/illustrations/NODE_001.png" class="node-image" />
  
  Node prose begins here...
  ```

  **Generic/Markdown**: Standard markdown image syntax:
  ```markdown
  ![Node NODE_001 illustration](specs/[FEATURE_DIR]/assets/illustrations/NODE_001.png)
  
  Node prose begins here...
  ```

- If no matching image file is found for a node, skip silently (no error, no placeholder)
- Report summary of embedded images:
  ```
  ✓ Embedded N images out of N nodes
  ```

### 3. Engine-Specific Compilation

#### Twine/SugarCube (twee → HTML via tweego)

**Compiler**: `tweego.exe` (from Twine installation)

**Process**:
1. Create `specs/<specname>/output/sugarcube/` directory
2. Read all source files (`.twee` or `.md`)
3. Convert `[MECHANIC:...]` tokens to SugarCube macro syntax
4. For `.md` files: strip YAML front matter, convert markdown choices to Twee wiki-links, wrap in `:: NODE_ID [header]` passage headers
5. Concatenate all processed content into a single `compile.twee` file
6. Run postprocessing scripts on `compile.twee` (inject headers, animations, theme overrides). See **Postprocessing** section below.
7. Include theme CSS (`story.css`) if present
8. Run `tweego.exe` with `compile.twee` (and `story.css` / `story.js`) as input
9. Generate `specs/<specname>/output/sugarcube/story.html`

**Output**: Playable HTML file - open in browser to play

**Retry logic**: 
- If tweego fails, attempts to auto-fix common issues (unmatched braces, etc.)
- Retries up to 3 times with error feedback

#### Ink (ink → JSON + HTML via inklecate)

**Compiler**: `inklecate.exe` (from Ink installation)

**Process**:
1. Create `specs/<specname>/output/ink/` directory
2. Validate with inklecate (-p flag):
   - Checks for syntax errors
   - Validates knot/stitch references
   - Reports validation errors
3. If validation passes, compile with inklecate (-c flag):
   - Generates `story.json` (compiled story format)
   - Generates `story.html` (HTML wrapper for story.json)
4. HTML wrapper includes:
   - Story metadata and info
   - Links to Ink web player and Inky IDE
   - Embedded story JSON for reference

**Output files**:
- `specs/<specname>/output/ink/story.json` - Compiled story (use with Ink runtime)
- `specs/<specname>/output/ink/story.html` - HTML display wrapper

**Retry logic**:
- If inklecate fails, attempts to auto-fix common divert syntax errors
- Retries up to 3 times with error feedback

#### Multi-Engine Compilation

When using `--all-engines`:
1. Reads export_engines from constitution.md or spec.yml
2. Iterates through each configured engine
3. Compiles each with its appropriate compiler
4. Reports success/failure for each engine
5. Exit code is 0 only if ALL engines succeed

Example:
```bash
speckit.compile --all-engines
# Compiles: sugarcube → tweego, ink → inklecate, etc.
# Outputs results for each engine
```

## Error Handling

**If compilation fails**:
- Shows specific error from compiler (first 10 lines)
- Attempts automatic fix for common errors
- Retries up to 3 times
- Shows which line numbers have issues (when possible)
- Offers next steps (re-draft nodes, check configuration, etc.)

**Example error output**:
```
❌ Compilation error:
   NODE-001.twee:15: Error: Unmatched '<<'
   NODE-003.twee:8: Error: Undefined variable: $player_name

🔧 Attempting auto-fix...
   Detected: Unmatched macro braces
   Fixed: NODE-001.twee

▶️  Attempt 2/3...
❌ Compilation failed (still has errors)

❌ Compilation failed after 3 attempts
   Check the errors above and fix the source files
```

## Runtime Testing & Walkthrough (SugarCube)

After successful compilation, `speckit.compile` runs a **runtime validation session** to catch errors that only appear when the game runs.

### Phase 1 — Quick Runtime Test (always runs unless `--no-test` or `--no-walkthrough`)

1. **Start browser session**: Open compiled `.html` in headless browser
2. **Walk starting passage**: Load initial node, verify:
   - No console errors
   - All UI elements render (HUD, menus visible)
   - Variables initialized correctly
3. **Test navigation**: Click each menu button to verify:
   - QuestUI, CharacterSheet, InventoryUI pages load without errors
   - Return to main passage works
4. **Walk key passages**: Auto-click a sample choice path (1-2 passages deep):
   - Execute conditionals (`<<if>>` blocks)
   - Test widget calls (`<<questList>>`, `<<inventory>>`, etc.)
   - Capture any runtime errors
5. **Check console**: Report any JavaScript errors that occurred during test
6. **Report results**: List any runtime issues found

### Phase 2 — Full Story Walkthrough (with `--walkthrough`)

When `--walkthrough` is passed, after Phase 1 passes, `speckit.compile` runs a **deep Playwright walkthrough** (`scripts/tests/full-story-walkthrough.spec.ts`) that explores the entire story graph:

1. **First-choice path**: Clicks the first available choice at each passage until reaching an ending (or depth limit of 20)
2. **All branches at depth 1**: Reloads and tries every top-level choice, verifying each leads to a valid passage
3. **BFS traversal**: Breadth-first search through all reachable passages (up to 50 nodes), verifying:
   - No passage contains `[error]` or `undefined` text
   - Every choice leads to a valid target passage
   - No JavaScript console errors

The walkthrough uses Playwright with Chromium headless and reports failures with passage-level detail:

```
🧪 Full-story walkthrough (--walkthrough)...
  ✓ First-choice path: 12 passages walked
  ✓ All branches at depth 1: 5/5 choices valid
  ✓ BFS traversal: 24 nodes visited, 0 errors
  ✓ No console errors detected

✅ Walkthrough passed
```

### Skipping Testing

```bash
speckit.compile --no-test       # Skip Phase 1 (quick runtime)
speckit.compile --no-walkthrough  # Skip both phases
speckit.compile --dry-run       # Validate only, don't write output
```

## Postprocessing

Postprocessing scripts in `specs/[FEATURE_DIR]/postprocessing/*.py` run **twice** during
compilation:

1. **Before compilation** (`stage="compile"`): Scripts can transform the combined source
   — inject passage headers/footers, add CSS animations, convert remaining tokens, or
   apply structural changes to `compile.twee` before `tweego.exe` processes it.

2. **After compilation** (`stage="post-compile"`): Scripts can modify the compiled output —
   replace CSS themes in `story.html`, inject analytics, or post-process the final HTML.

Each script receives a `postprocess(ctx)` call with:
- `spec_path` — spec directory
- `engine` — "sugarcube" or "ink"
- `stage` — `"compile"` (pre-compile) or `"post-compile"` (post-compile)
- `source_dir` — output directory (contains `compile.twee` before compilation, `story.html` after)
- `output_dir` — output directory

Scripts that only modify source should check `stage == "compile"` or be idempotent.
Scripts that modify the compiled HTML should check `stage == "post-compile"`.

To generate postprocessing scripts, use `speckit.postprocessing`.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| "No drafted nodes found" | No files in `specs/[FEATURE_DIR]/draft/<ENGINE>/` | Run `speckit.implement` first |
| "YAML validation failed" | Malformed front matter | Check node header syntax |
| "Unknown hook type" | Invalid [MECHANIC:...] block | Check hook type against mechanics.md |
| "Variable undefined" | Export name missing in `variables.md` | Add export name for target engine |
| "tweego.exe not found" | Twine toolchain not installed | Use template-based HTML generation or install tweego |
