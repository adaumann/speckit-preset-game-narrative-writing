# The Shrine: A Narrative Design Sample

A complete, playable SugarCube 2.30.0 interactive fiction game demonstrating **Tier 2 game narrative mechanics** (optional systems with moderate complexity): quest tracking, inventory management, and character attribute display.

**File**: `shrine.html` (standalone, ~476 KB) — open in any web browser to play.

**What this teaches**: SugarCube widget patterns, Twee markup, quest/inventory system implementation, CSS styling for interactive fiction.

---

## Quick Start

1. **Open in browser**: Double-click `shrine.html` or open via File → Open
2. **Click menu buttons** (Character, Inventory, Quests) to see UI in action
3. **Play through** the 5–10 minute narrative (3 dialogue branches → 6-stage quest)
4. **View the source**: Read `shrine-widgets.twee`, `shrine-ui.twee`, `shrine-nodes.twee` to understand implementation

---

## What This Demonstrates

### Core Mechanics (Tier 2)

1. **Quest System**
   - Multi-stage tracking: `$quest_[id]_status` (active/completed/failed) + `$quest_[id]_stage` (integer)
   - Advancement via `<<advanceQuestStage>>` widget, display via `<<questList>>` widget
   - Visual: Active (gold), Completed (green), Failed (red)

2. **Inventory System**
   - Array-based: `$inventory = [{key: "name", qty: count}, ...]`
   - Pickup: `<<pickupItem "key" "Display Name">>` (adds or increments)
   - Display: `<<inventory>>` widget shows all items with quantities

3. **Character Profile**
   - Attributes: `$character = {name, intelligence, wisdom, power, gold}`
   - Display: `<<charProfile>>` widget renders as HTML table
   - State changes: Modifiable via `$character.attribute += N`

### Game Structure

- **8 core passages** (Start, AskHistory, AfterHistory, Donate, Donate5, Donate1, Search, End, Final)
- **Multi-branch narrative**: Player choices lead to 3 different paths with different quest progression
- **Intelligence gate**: "Ask History" only available if `$character.intelligence >= 5`
- **Donation state**: Tracks donation amount, triggers different consequences
- **6-stage quest progression**: Tracks player advancement through all paths

## Key Files & What They Do

### shrine-widgets.twee

**Widget definitions** (reusable macro functions):

| Widget | Purpose |
|--------|--------|
| `<<questList>>` | Render all quests grouped by status (active/completed/failed) |
| `<<questProgress "id">>` | Show single quest status inline: "📍 Stage N (In Progress)" |
| `<<advanceQuestStage "id">>` | Increment quest stage counter by 1 |
| `<<completeQuest "id">>` | Mark quest as completed, display message |
| `<<charProfile>>` | Render character sheet as table with attributes |
| `<<inventory>>` | List all inventory items with quantities |
| `<<pickupItem "key" "Name">>` | Add item to inventory or increment qty |
| `<<hasItem "key">>` | Check if item exists in inventory |

**Critical pattern**: Widget definitions must be wrapped in a passage with `[widget]` tag:
```twee
:: ShrineWidgets [widget]
<<widget "questList">>
  [widget body here]
<</widget>>
```

### shrine-init.twee

**StoryInit passage** — initializes all game state when story starts:

```twee
:: StoryInit
  $character = {
    name: "The Wanderer",
    intelligence: 5, wisdom: 6, power: 4, gold: 10
  }
  $inventory = []
  $quest_shrine_blessing_status = "active"
  $quest_shrine_blessing_stage = 1
  [... other quest/variable initialization ...]
```

### shrine-ui.twee

**UI Structure**:
- **StoryCaption**: HUD showing gold count & item count
- **StoryMenu**: Navigation buttons (Character, Inventory, Quests, Home, Saves, Restart)
- **CharacterSheet/InventoryUI/QuestUI**: Menu pages
- **StoryStylesheet**: ~200 lines of CSS (dark theme with purple accents)

**Three-theme CSS system** (select by editing CSS variables):
- Dark theme: `#a78bfa` accents, `#0f0f13` background
- Light theme: `#8b6914` accents, `#f5f5ef` background
- Minimal theme: Text-only, no colors

### shrine-nodes.twee

**Narrative passages**:
- **Start**: Entry point, three dialogue choices
- **AskHistory**: Intelligence-gated scene (requires `$character.intelligence >= 5`)
- **AfterHistory**: Consequence of history choice
- **Donate**: Donation scene with two path branches
- **Donate5/Donate1**: Donation amount consequences
- **Search**: Treasure search scene with inventory pickup
- **End**: Quest completion
- **Final**: Victory screen

## SugarCube Patterns

### 1. Widget Definition Syntax

```twee
:: WidgetName [widget]
<<widget "questList">>
  <<set _var = "something">>
  <div>
    <<= _var>>
  </div>
<</widget>>
```

**Key points**:
- Widget passage must have `[widget]` tag
- Widget names are lowercase, called as `<<widgetName>>`
- Variables prefixed with `_` are temporary (scoped to widget)
- Variables prefixed with `$` are persistent (state)

### 2. Suppressing Whitespace in Widget Output

**Problem**: SugarCube converts every newline in passage content to `<br>` or block-level whitespace. Multi-line `<<for>>` loops generate huge gaps.

**Solution 1 — `<<silently>>` block**:
```twee
<<silently>>
  <<for _key, _val range State.variables>>
    <<if _key.startsWith("quest_")>>
      [loop body]
    <</if>>
  <</for>>
<</silently>>
```

**Solution 2 — `[nobr]` passage tag**:
```twee
:: QuestUI [nobr]
<h2>Quests</h2><<silently>>
  [all computation here]
<</silently>><<if _html>>[output]<</if>>
```

The `[nobr]` tag suppresses newline-to-break conversion for the entire passage, preventing gaps between elements.

### 3. Loop Syntax

**SugarCube 2.x supports these loop forms**:

✅ **for...range** (object iteration):
```twee
<<for _key, _val range State.variables>>
  <<= _key>>: <<= _val>>
<</for>>
```

✅ **for** (numeric loop):
```twee
<<for _i = 0; _i < 10; _i++>>
  Item <<= _i>>
<</for>>
```

❌ **for...of** (NOT supported in SugarCube 2.x):
```twee
<<for _item of $inventory>>  [← ERROR]
```

### 4. Conditional Rendering

```twee
<<if $character.intelligence >= 5>>
  [shown when smart enough]
<</if>>

<<if $inventory.length > 0>>
  [shown when carrying items]
<</if>>

<<switch $quest_shrine_blessing_status>>
  <<case "active">>
    [shown for active quests]
  <<case "completed">>
    [shown for completed quests]
  <<default>>
    [fallback]
<</switch>>
```

### 5. String Building Pattern

When output needs conditional elements, build HTML as string then print once:

```twee
<<set _html = "">>
<<if $condition1>>
  <<set _html += "<h3>Title</h3>">>
<</if>>
<<for _item in $inventory>>
  <<set _html += "<li>" + _item.name + "</li>">>
<</for>>
<<print _html>>
```

This avoids rendering empty tags when conditions are false.

### 6. Temporary Variable Naming

Prefix temporary variables with `_` to indicate they're not state:
- `_i`, `_key`, `_val`: Loop counters/iteration
- `_html`, `_text`: Temporary strings
- `_qname`, `_qstage`: Temporary quest data

These are scoped to the current passage/widget and don't persist.

### 7. Navigation Links

```twee
<<link "Back" "Start">><</link>>       [← Jump to Start passage]
<<link "Next" previous()>><</link>>    [← Use previous() for history]
```

**Note**: `previous()` may not work in all contexts. Use explicit passage name instead:
```twee
<<link "Back" "StartPage">><</link>>   [← Always works]
```

## How to Use as a Reference

### For Quest System

1. Copy the quest mechanics from **shrine-widgets.twee** (`<<questList>>`, `<<advanceQuestStage>>`, `<<completeQuest>>`)
2. Initialize quest variables in your **StoryInit** following the pattern in **shrine-init.twee**
3. Use the CSS from **shrine-ui.twee** for styling (`.quest-item.active`, `.quest-item.completed`, etc.)

### For Inventory System

1. Use `<<pickupItem>>` and `<<hasItem>>` widgets as-is
2. Initialize `$inventory = []` in StoryInit
3. Reference the `<<inventory>>` widget UI pattern

### For Character Stats

1. Copy the `<<charProfile>>` widget
2. Modify the `$character` object structure to match your game's attributes
3. Update the profile table HTML to match your attribute names

### For Whitespace Issues

If you encounter large gaps in your widget output:
1. Wrap loop logic in `<<silently>>...<</silently>>`
2. Add `[nobr]` tag to passage header: `:: PassageName [nobr]`
3. Concatenate output to strings and use `<<print>>` once

## Known Limitations

1. **No persistent storage** — Saves are browser-local only (lost on cache clear)
2. **Single POV** — Game uses second-person throughout (easy to change)
3. **Tier 2 mechanics only** — No skill checks, no dynamic NPC dialogue trees, no complex state machines
4. **No external files** — Everything is in one HTML file (easy to distribute, hard to edit)

## Extending This Project

### Add a New Quest

1. Add quest initialization to **StoryInit**:
   ```twee
   $quest_myquest_status = "active"
   $quest_myquest_stage = 1
   ```

2. Add advancement logic to narrative passage:
   ```twee
   <<advanceQuestStage "myquest">>
   ```

3. Mark as completed when done:
   ```twee
   <<completeQuest "myquest">>
   ```

### Add Character Attributes

1. Add to `$character` in **StoryInit**:
   ```javascript
   $character = {...existing, charisma: 7}
   ```

2. Update `<<charProfile>>` widget to include new row:
   ```html
   <tr><td>Charisma</td><td><<= $character.charisma>></td></tr>
   ```

### Switch to Light Theme

Edit **StoryStylesheet** in shrine-ui.twee:
- Change `background: #2a2a2a` to `background: #f5f5ef`
- Change `color: #e0e0e0` to `color: #333`
- Update accent color `#8eb4d4` to `#8b6914`
